# Welcome to Domain-Driven Design

This skill provides a structured workflow for discovering your domain model through **Event Storming** and translating it into an implementation-ready Dev Spec.

## What is Event Storming?

Event Storming is a workshop-based method for exploring complex business domains. Instead of starting with requirements, we start with **domain events** — things that happened in the past tense:

- "Video Commissioned"
- "Script Approved"
- "Asset Rejected"
- "Render Completed"

By mapping out what happens, we discover:
- **Commands** — what triggers events ("Approve Script")
- **Policies** — automatic reactions ("When X, then Y")
- **Aggregates** — things that hold state (Video, Script, Asset)
- **Read Models** — views actors need to make decisions

The result is a **Domain Model** — a complete picture of how your system works.

## The Three-Stage Workflow

### 1. `/ddd begin` — Discover (Interactive)
I'll guide you through 8 stages of event storming:
1. Domain Context (scope, actors, problem)
2. Events Brainstorm (chaotic, no structure yet)
3. Events Timeline (organize, number E-01, E-02...)
4. Commands (what triggers events? C-01, C-02...)
5. Actors (who issues commands?)
6. Policies (automatic reactions, P-01, P-02...)
7. Aggregates (state holders)
8. Read Models (decision-support views)

**Output:** `docs/SKETCHBOOK.md` (raw notes, informal)

**My role:** I wear two hats — domain expert partner (asking Socratic questions) and facilitator (recording, structuring).

### 2. `/ddd draft` — Formalize
I read your sketchbook and produce a formal Domain Model document following the template structure. This includes:
- Numbered events, commands, policies
- State machines for aggregates
- Traceability (policies reference events, commands reference actors)
- DDD→Dev Spec translation map

**Output:** `docs/DOMAIN-MODEL.md` (formal, canonical)

### 3. `/ddd accept` — Translate
I apply the DDD→Dev Spec protocol to generate an implementation-ready Dev Spec:
- Actors → Personas (Section 1.4)
- Aggregates → Data Model (Section 5.1)
- Policies → Requirements (Section 3, EARS format)
- Commands → Operational Flows (Section 4)
- Read Models → API/UI Design (Section 5)
- Aggregates + Policies → Implementation Stories (Section 8)

**Output:** `docs/[project]-devspec.md` (implementation spec)

After this, your Dev Spec feeds into existing workflow: `/prepwaves` → `/nextwave` → execution.

## When to Use DDD vs Requirements Engineering

| Approach | Best For | Starts With |
|----------|----------|-------------|
| **DDD (Event Storming)** | Complex workflows, state machines, multi-actor coordination | Domain events ("what happened?") |
| **Requirements Engineering** | Well-understood problems, API design, technical solutions | Goals, constraints, user needs |

Both converge at the Dev Spec. Choose based on problem characteristics.

**DDD shines when:**
- The workflow is complex (many decision points, loops, parallel paths)
- Multiple actors coordinate (human + agent, or multiple teams)
- State transitions are critical (lifecycle management, approval gates)
- You're not sure what the requirements are yet (discovery needed)

**Requirements Engineering works when:**
- The problem is well-understood (you already know what to build)
- The solution is technical (API design, data pipeline, infrastructure)
- The workflow is simple (linear, few branches)

## The Socratic Method

During `/ddd begin`, I'll use the **Socratic method** — asking questions rather than dictating answers. Expect:

- **"What happens after X?"** — discovering follow-on events
- **"Who decides that?"** — identifying actors and commands
- **"What if Y fails?"** — uncovering error paths and policies
- **"How do you know when Z is complete?"** — defining exit conditions
- **"Does this always happen, or only sometimes?"** — distinguishing commands from policies

The best domain models emerge from **questioning**, not dictation. I'm here to help you discover what you already know.
